Product Requirements Document (PRD)
Project Aegis: The Sovereign AI Passport

1. Product Overview & Problem Statement
1.1 Abstract
Project Aegis delivers a decoupled, provider-agnostic context layer ("AI Passport") that abstracts a user’s long-term memory, custom functional capabilities, and local configurations away from proprietary AI vendor silos. The system acts as a standard interoperable gateway, allowing users to point any AI model client (local terminal, IDE, or cloud orchestration harness) to a single secure vault without reconfiguration.
1.2 The ProblemAs of 2026, the Agentic AI market is highly fragmented. A developer's system instructions, tools, and vector knowledge indices are deeply coupled with their respective client runtimes (e.g., Cursor, Claude Code, or specialized terminal CLIs). Shifting providers requires manual migration and rebuilding of state. Furthermore, enterprise adoption is throttled by privacy constraints: default cloud provider behaviors retain telemetry and prompts for 7–30 days for abuse monitoring, representing a severe corporate liability.
1.3 The SolutionA unified Git-style folder structure governed by open industry specifications (Model Context Protocol, AgentSkills, and LLM Wiki v2).Free Tier: Runs locally via a zero-cost Docker container on localhost.Paid Tier: Deploys onto serverless, hardware-encrypted Trusted Execution Environments (TEEs) that scale to zero, establishing a zero-trust, cryptographically isolated boundary with sub-second initialization times.

2. Core Product PrinciplesTotal Sovereign Portability: The user owns the raw state. The compute engine (the LLM) is a completely hot-swappable utility.Hardware-Enforced Zero Trust: Privacy guarantees are backed by silicon physics (AMD SEV-SNP/Intel TDX), not software configurations or cloud provider privacy policies.Developer Experience First: Operational network and security wrappers must execute asynchronously, adding $<5\text{ms}$ overhead to the active prompting loop.Zero Blockchain Bloat: No public ledger, gas fees, or token economics. Native web standards (WebAuthn/Passkeys) provide authentication.

4. Functional Requirements
4.1 Interface & Transport Layer (Model Context Protocol)FR-1.1: The system must expose all features using the Model Context Protocol (MCP) standard over JSON-RPC 2.0.FR-1.2: The local client adapter must accept stdio pipes, while the cloud TEE platform must expose standard Streamable HTTP endpoints using Server-Sent Events (SSE).FR-1.3: Progressive Disclosure Enforcement: At session initialization (tools/list), the server must only transmit shallow tool metadata (name, intent description). The complete execution logic and referenced assets must only hydrate into the context window when actively requested via a tools/call invocation.
4.2 The Capability Specification (AgentSkills)FR-2.1: The repository filesystem must look for an executable path matching the agentskills.io open specification layout under /skills/.FR-2.2: Each unique capability must be contained in its own subdirectory containing a mandatory SKILL.md (housing JSON Schema descriptors and imperative natural language instructions limited to $<500$ lines) and optional /scripts or /references directories.
4.3 The Memory & Context Specification (LLM Wiki v2)FR-3.1: Long-term memory and structured knowledge must be formatted using plain-text markdown documents inside the /wiki/pages/ directory utilizing cross-linked [[wiki-links]].FR-3.2: Contradiction Mitigation: The ingestion pipeline must parse inbound context updates against existing wiki pages. If an incoming parameter explicitly opposes an old record, the system must write the anomaly to a central _contradictions.md file and adjust the confidence metrics in the page's YAML frontmatter.
4.4 Identity & Authentication Layer (Passkeys)FR-4.1: Authentication must be completely passwordless and seed-phrase-free, routing natively through the WebAuthn API (Passkeys via TouchID/FaceID).FR-4.2: On validation, the client harness must transmit a short-lived Private Key JWT assertion via the HTTP header. The TEE must evaluate this signature against the user's bound public key held in memory before decrypting the master data volume.
4.5 The Security Firewall (Zero Data Retention Enforcement)FR-5.1: The TEE gateway proxy must intercept all outgoing LLM provider API requests.FR-5.2: The gateway must append vendor-specific programmatic non-retention flags (e.g., store: false or headers forcing Zero Data Retention) before payload routing.FR-5.3: If an outgoing call targets an unverified consumer endpoint that lacks contractual ZDR safety guarantees, the gateway must drop the connection and throw an alert to the client interface.
4.6 The Absolute Erasure Kill-Switch (Right to Forgotten)FR-6.1: To comply with GDPR Article 17, the application must feature a primary "Nuke Switch" executable from both the dashboard and local CLI tool.FR-6.2: Crypto-Shredding Engine: On execution, the system must permanently delete the Master Key Encryption Key ($KEK$) directly from the TEE's hardware secure volatile memory management block.FR-6.3: Following the $KEK$ wipe, the system must send asynchronous bulk delete calls to clear the encrypted storage blocks on Cloudflare R2, leaving all legacy backups as unrecoverable mathematical noise.

5. Non-Functional Requirements & Technical Constraints
5.1 Latency Budget BlueprintTo safeguard the developer experience, the system must strictly balance its performance parameters:TEE Cold Start Latency: Container initialization inside the hardware enclave from an unhydrated state to live HTTP/SSE availability must finish in $\le 150\text{ms}$.Attestation Signature Time: Generating and locally parsing the CPU-signed cryptographic Remote Attestation report must take $\le 300\text{ms}$ at session initialization.
5.2 Storage SecurityNFR-2.1: All files synchronized to Cloudflare R2 must be encrypted client-side using authenticated AES-256-GCM envelope encryption before network transit.NFR-2.2: The runtime cloud container must implement strict Trusted Computing Base (TCB) version parsing, automatically refusing execution if the underlying CPU hardware firmware fails compliance audits for active microcode side-channel patches.
